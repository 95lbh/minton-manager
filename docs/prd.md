## 1. 서비스 정의

### 한 줄 정의
배드민턴 동호회 관리자가 **출석 → 코트 배정 → 게임 진행 → 기록/통계**를 하나의 운영 흐름에서 빠르게 처리하는 **멀티테넌트 웹앱**. 별도의 **대회 모드**로 토너먼트 운영(대진표·승패·점수)도 지원한다.

### 두 가지 운영 모드
- **일반 운영 모드 (기본)**: 평소 동호회 게임 운영. **승패/점수는 기록하지 않는다.** 게임 횟수와 "누구와 게임했는지(파트너/상대)"만 남긴다 — 기록 부담 최소화.
- **대회 모드 (별도)**: 대회 참가자 등록, **승패·점수 기록**, 토너먼트/대진표 작성(수동·자동), 대회 결과·기록 관리. 일반 모드와 데이터·화면을 분리한다.

### 접근/배포
URL만 알면 **어떤 기기·브라우저·OS**에서든 인터넷을 통해 접속하는 **웹 서비스**. **PWA**로 설치해 앱처럼도 사용 가능(오프라인 캐시는 3단계).

### 제품 방향성 (가장 중요)
- 회원은 **최대한 아무것도 하지 않아도 된다.** 조작 주체는 관리자다.
- 관리자가 **출석과 게임을 빠르게** 관리할 수 있어야 한다.
- **코트 상황이 한눈에** 보여야 한다.
- 운영 중 **실수가 적어야** 한다 (중복 배정·기록 유실 방지).
- **모바일에서도** 편하게 조작할 수 있어야 한다.
- 데이터가 쌓이면 **통계·자동 배정**으로 확장한다.

> UX 기준: "멋진 앱"이 아니라 "현장에서 빠르게 쓰는 운영 도구".

## 2. 주요 사용자

**로그인하는 사용자는 운영 주체(운영자/스태프)뿐이다.** 회원은 로그인하지 않으며 관리자가 관리하는 데이터로만 존재한다.

| 역할 | 로그인 | 설명 | MVP 우선순위 |
|---|---|---|---|
| **Club Admin (클럽 운영자)** | ✅ | 클럽 생성·회원 등록·role 부여, 출석·배정·게임·대회 전반 | **최우선 (MVP 중심)** |
| Staff (스태프) | ✅ | 운영자가 권한 부여. 출석/배정 등 운영 보조 | 후순위 |
| Member (회원) | ❌ | 출석 대상이자 통계 주체. **로그인 안 함, 직접 조작 없음** | 데이터 객체 중심 |
| Super Admin | ✅ | 다중 클럽 / 시스템 관리 | 3단계 |

- 권한 구조: `Super Admin > Club Admin > Staff`. MVP는 **Club Admin** 권한 하나로 운영 가능하게 구현한다.
- **클럽 운영자가 클럽을 만들면** 그 클럽에 회원을 받고, 로그인 사용자에게 **스태프 등 role을 부여**할 수 있다.
- 한 사람이 여러 클럽 회원이어도 무방(전역 사용자 마스터 불필요).
- **비회원 일회성 운영**: 계정 없이도 서비스를 일시적으로 운영 가능(임시 클럽/세션). 데이터 영속·통계는 제한적.

## 3. 핵심 사용자 시나리오

1. **운영 시작** — 관리자가 로그인 → 클럽 선택 → "오늘의 출석 세션" 생성.
2. **출석 처리** — 회원 검색/체크로 출석 등록. (MVP: 관리자 직접 체크 / 2단계: QR 스캔) 비회원·신규는 **임시 출석자(게스트)** 로 등록.
3. **코트 배정** — 출석자 풀에서 코트에 4명 배정(드래그앤드롭 또는 버튼). 중복 배정·미출석 배정을 차단. (2단계: 규칙 기반 자동 추천)
4. **게임 진행** — 게임 시작 → 종료. 종료 시 **참가 멤버 / 파트너 / 상대 / 시간**을 기록 저장. 종료된 회원은 대기열로 복귀.
5. **다음 게임 배정** — 오래 쉰 회원 우선으로 다음 코트 배정.
6. **통계 확인** — 회원별 출석/게임 횟수, 클럽별 일자 통계 조회.

## 4. 기능 요구사항

### 4.1 코트 배정 및 게임 관리 (핵심)
- 코트 목록 관리 (등록/상태)
- 출석자 풀 확인 및 검색
- 코트에 회원 배정 (드래그앤드롭 / 버튼)
- 코트별 현재 게임 멤버 표시
- 게임 시작 / 종료
- 대기 회원 / 휴식 회원 관리
- **중복 배정 방지** (한 회원이 동시에 두 코트에 배정 불가)
- 파트너·상대 조합 기록
- 게임 종료 후 다음 게임 배정 지원

### 4.2 자동 배정 추천 (2단계, 확장 설계)
초기에는 단순 규칙 기반, 추후 AI 추천으로 확장 가능하게 인터페이스를 분리한다.
- 오래 쉰 회원 우선
- 최근 많이 뛴 회원 후순위
- 같은 파트너 반복 회피
- 실력 레벨 균형
- 혼복 여부 선택적 고려

### 4.3 로그인 / 인증 (부가)
- **SSO = Google OAuth** (Supabase Auth). 운영자/스태프만 로그인.
- 클럽별 데이터 분리, 관리자 인증. 운영자가 스태프 role 부여.
- **비회원 일회성 운영** 경로 별도 제공(로그인 없이 임시 운영).
- MVP는 **Club Admin 중심**

### 4.4 출석 체크 (부가)
- MVP: 관리자 직접 체크 / 회원 검색 후 출석 처리 / 모바일 간편 처리
- 2단계: QR 출석 (관리자가 오늘의 QR 생성 → 회원 스캔 → 식별/출석)
- **회원은 로그인 없이 출석 가능**한 구조 우선

### 4.5 통계 (부가)
- 회원별: 출석/게임 참여 횟수, 평균 대기시간, 파트너·상대 이력, 최근 참여일, 참여율
- 클럽별: 일자별 출석/게임 수, 코트 사용률, 시간대 혼잡도, 장기 미참여/단골 회원
- 초기에는 테이블·카드 UI, 이후 차트 확장
- (일반 모드는 승패가 없으므로 승률 통계는 **대회 모드 한정**)

### 4.6 대회 모드 (별도 모드)
일반 운영 모드와 분리된 토너먼트 운영 도구. 일반 모드의 회원 데이터를 가져오거나 대회 전용 참가자를 따로 등록할 수 있다.
- 대회 생성 / 참가자 등록 (기존 회원 불러오기 또는 신규 입력)
- 대진표 작성: **수동 편성** 및 **자동 편성**(시드/랜덤/리그·토너먼트)
- 경기별 **승패·점수 기록**
- 대회 진행 현황(라운드/브래킷) 시각화
- 대회 결과·순위·기록 저장 및 조회
- 형식: 토너먼트(싱글/더블 엘리미네이션), 리그(조별) 등 단계적 확장

## 5. MVP 범위 (단계별)

### MVP 1단계 — 필수 (현재 목표)
- [ ] 관리자 로그인
- [ ] 클럽 생성 / 선택
- [ ] 회원 등록 / 수정 / 비활성화
- [ ] 코트 등록
- [ ] 오늘의 출석 세션 / 출석 체크
- [ ] 코트별 회원 배정 (중복 방지)
- [ ] 게임 시작 / 종료
- [ ] 게임 기록 저장
- [ ] 회원별 출석 횟수 / 게임 횟수 확인

### MVP 2단계
- QR 출석 체크
- 규칙 기반 자동 코트 배정 추천
- 회원별 / 클럽별 통계
- 모바일 최적화 개선
- **대회 모드 1차**: 대회 생성·참가자 등록·수동 대진표·승패/점수 기록

### MVP 3단계
- SSO 다양화 · PWA · 오프라인 캐시 · 동기화 큐
- 고급 통계 · AI 기반 배정 추천 · 알림
- **대회 모드 확장**: 자동 대진 편성(시드/리그/엘리미네이션)·브래킷 시각화·대회 순위/기록

> 참고: **PWA(웹앱 설치)** 는 3단계지만, 기본 웹 접근(어떤 기기·브라우저든 URL 접속)은 1단계부터 보장된다(반응형 웹).

## 6. 사용자 플로우

### 6.1 전체 흐름
```
로그인 → 클럽 선택 → [대시보드]
  ├─ 출석 관리: 세션 생성 → 회원 검색/체크 → 출석자 풀 확정
  ├─ 코트 배정: 출석자 풀 → 코트 N에 4명 배정 → 게임 시작
  │     └─ 게임 종료 → 결과 기록 → 회원 대기열 복귀 → 다음 배정
  ├─ 회원 관리: 등록 / 수정 / 비활성화
  └─ 통계: 회원별 / 클럽별 조회
```

### 6.2 코트 배정 화면 (운영 중 핵심 화면)
```
[ 출석자 풀 / 대기열 ]        [ 코트 1 ] [ 코트 2 ] [ 코트 3 ] ...
  - 김OO (대기 12분)            ┌────────┐ ┌────────┐
  - 이OO (대기 8분)             │ A · B  │ │ (비어) │
  - 박OO (게임중) ✗             │ vs     │ │        │
  - ...                         │ C · D  │ │        │
                                │[종료]  │ │[배정]  │
                                └────────┘ └────────┘
```
- 실시간, 즉시 반응. 큰 버튼. 한눈에 코트 현황.
- 게임 중 회원은 출석자 풀에서 비활성/표시되어 중복 배정 차단.

## 7. 핵심 데이터 구조 (개념 모델)

> 상세 SQL / RLS / 인덱스는 `database.md`에서 확정. 여기서는 엔티티와 관계만 정의.

멀티테넌트 기본 단위 = **`club`**. 모든 운영 데이터는 `club_id`로 격리.

**일반 운영 데이터**

| 테이블 | 역할 | 핵심 컬럼 (개념) |
|---|---|---|
| `clubs` | 클럽 (테넌트) | id, name, owner_id, is_temporary(일회성 여부) |
| `auth_profiles` | **로그인 사용자(운영자/스태프)** ↔ 권한 | id(=auth user), club_id, role(admin/staff) |
| `club_members` | 회원 (로그인 안 함, 클럽 소속·레벨·성별) | id, club_id, name, level, gender, status |
| `courts` | 코트 | id, club_id, name, status |
| `attendance_sessions` | 일자별 출석 세션 | id, club_id, date, status |
| `attendance_records` | 세션 내 개인 출석 | id, session_id, member_id, is_guest |
| `games` | 코트별 게임 | id, club_id, court_id, session_id, started_at, ended_at, status |
| `game_players` | 게임 참가자 (파트너/상대 = team 구분) | id, game_id, member_id, team |
| `member_stats` | 집계 (view / MV 후보) | member_id, attend_cnt, game_cnt, last_played_at |
| `invitations` | 클럽 초대(스태프 등) | id, club_id, email, role |

> 일반 모드에는 **승패/점수 테이블이 없다.** `game_players`가 "누가 누구와 뛰었는지"를 담고, 게임 횟수는 `games`/집계로 계산한다. `users`(전역 마스터)는 두지 않고 `club_members`로 통합한다.

**대회 모드 데이터** (별도, 승패/점수 기록)

| 테이블 | 역할 | 핵심 컬럼 (개념) |
|---|---|---|
| `tournaments` | 대회 | id, club_id, name, format(단식/복식·토너먼트/리그), status |
| `tournament_participants` | 참가자 (회원 연결 또는 신규) | id, tournament_id, member_id(nullable), name, seed |
| `tournament_matches` | 경기/대진 | id, tournament_id, round, bracket_pos, team_a, team_b |
| `tournament_results` | 경기 결과 (**승패·점수**) | id, match_id, winner, score_a, score_b |

### 공통 컬럼 규칙
모든 운영 테이블: `id, club_id, created_at, updated_at, deleted_at(soft delete), created_by, status`.

### 정합성 원칙
- 게임 기록·중복 배정 등 **정합성이 중요한 작업은 트랜잭션 + DB 제약**으로 보장.
- 한 세션 내 한 회원이 동시에 진행 중(`status=ongoing`)인 게임에 중복 참가 불가.
- 통계성 데이터는 **view 또는 materialized view** 고려.
- 삭제는 hard delete 대신 `deleted_at` 기반 **soft delete** 우선.

### 주요 관계
```
clubs 1─* auth_profiles (운영자/스태프)
clubs 1─* club_members (로그인 안 하는 회원)
clubs 1─* courts
clubs 1─* attendance_sessions 1─* attendance_records *─1 club_members
clubs 1─* games *─1 courts
games 1─* game_players *─1 club_members        # 일반 모드: 승패 없음

clubs 1─* tournaments 1─* tournament_participants *─0..1 club_members
tournaments 1─* tournament_matches 1─1 tournament_results   # 대회 모드: 승패·점수
```

## 8. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| 실시간 배정 중 동시성/중복 배정 | DB 유니크/체크 제약 + 낙관적 업데이트 롤백 |
| 게임 종료 전 기록 유실 | 종료를 트랜잭션으로 처리, 네트워크 실패 시 재시도 |

| 비회원/임시 출석자 정합성 | `is_guest` 플래그, 통계 산정 시 분리 처리 |
| 관리자 실수 (삭제/오배정) | 확인 모달 + soft delete + undo 고려 |

## 9. 확정된 의사결정 사항 (2026-05-30)

1. ✅ **SSO = Google OAuth** (Supabase Auth).
2. ✅ **회원은 로그인하지 않는다.** 운영자/스태프만 로그인하고, 회원은 관리자가 관리하는 `club_members` 레코드로만 존재.
3. ✅ **일반 모드는 승패/점수 기록 안 함** — 게임 횟수 + 파트너/상대만. **승패·점수는 대회 모드에서만** 기록.
4. ✅ **`users`(전역 마스터) 두지 않고 `club_members`로 통합.** 한 사람이 여러 클럽에 속해도 무방.

추가 확정:
- 웹 서비스(반응형, URL 접속) + **PWA** 지원.
- **비회원 일회성 운영** 경로 제공.
- **대회 모드**를 별도 모드로 분리(2단계 1차, 3단계 자동 대진 확장).
