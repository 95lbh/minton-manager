# QR 셀프 출석 설계

> 상태: 구현 중 v1 · 전제: [prd.md](prd.md) 4.4, [architecture.md](architecture.md)

## 흐름
1. 관리자: 출석 탭 → **QR 출석** 버튼 → 오늘 세션의 QR(또는 링크) 표시.
2. 회원: QR 스캔 → `/checkin/{token}` (로그인 불필요, 공개 페이지).
3. 셀프 출석 화면:
   - **이름 검색** → 결과(이름·성별·급수)에서 본인 선택 → 출석.
   - **처음 온 회원**: "신규 회원 등록" → 이름/성별/급수 입력 → 회원 생성 + 출석.
   - **게스트**: "게스트로 출석" → 이름/성별/급수(1회성) → 게스트 출석.

## 보안 (결정됨)
- **세션 토큰 + 공개 RPC**. 세션마다 추측 불가 토큰(`checkin_token`) 발급.
- 셀프 출석 화면은 RLS에 막히므로, 토큰을 인자로 받는 **SECURITY DEFINER 공개 RPC**로만 동작:
  - `checkin_session_info(token)` → 클럽명/세션상태(유효성)
  - `checkin_search_members(token, q)` → 그 클럽 회원 중 이름 검색 결과(id, name, gender, level, 출석여부)
  - `checkin_member(token, member_id)` → 회원 출석
  - `checkin_new_member(token, name, gender, level)` → 신규 회원 생성 + 출석
  - `checkin_guest(token, name, gender, level)` → 게스트 출석
- 토큰은 **그날 세션만** 유효(세션이 곧 날짜). 세션 status=closed면 거부.
- 익명 호출이므로 입력은 함수 내부에서 검증(이름 필수, 길이 제한, 토큰 일치).

## 동명이인 처리
- 검색 결과에 **성별·급수 병기**로 구분.
- 이미 출석한 사람은 "출석됨"으로 표시·비활성.
- 성별·급수까지 같은 완전 동일건은 드묾 → MVP는 그대로 노출(관리자 사후 정리). 추후 가입순 번호 부여 고려.

## 노출 최소화
- 전체 명단을 한 번에 주지 않고 **검색어 있을 때만** 결과 반환(최소 1~2글자). 결과 수 제한(예: 20).

## 미적용/추후
- 토큰 회전(재발급), QR 이미지 다운로드, 출석 마감 후 안내 페이지 등.
